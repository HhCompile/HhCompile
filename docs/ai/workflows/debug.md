# 调试工作流

> 系统性 Debug 方法

## 🎯 适用场景

- 复杂 bug 定位（无法通过 log 找到）
- 偶发问题（难以复现）
- 性能问题根因分析

## 📋 4 阶段流程

### 1️⃣ Reproduce（复现）
- 写最小失败用例
- 明确"何时出现 / 何时不出现"
- 记录输入、环境、操作步骤

### 2️⃣ Hypothesize（假设）
- 列可能根因（**至少 3 个**）
- 排优先级：高频 → 低频
- 排除明显不相关的

### 3️⃣ Verify hypothesis（验证）
- 逐个假设验证
- 添加针对性 log
- 用 debugger 单步

### 4️⃣ Fix root cause（修根因）
- 不修症状
- 写"反向测试"（确保 bug 不再出现）

## 💬 提示词模板

```
这是一个 bug 报告：
[bug 现象]

[已尝试的方法]

[相关代码]

请帮我：
1. 列 3 个可能根因
2. 每个根因的验证方法
3. 推荐的修复方案
```

## ⚠️ 反模式

- ❌ 看到错误就改 try/catch 吞掉
- ❌ 加 `console.log` 满屏撒网
- ❌ 不复现就修
- ❌ 修改症状而非根因

## 📊 实际收益

- 调试时间：~2h → ~30min
- 复发率：< 5%
