# TDD 工作流

> RED → GREEN → IMPROVE

## 🎯 适用场景

- 新功能开发
- Bug 修复
- 重构（行为不变）

## 📋 流程

```
🔴 RED：先写测试（必须失败）
🟢 GREEN：写最小实现（让测试通过）
🔵 IMPROVE：重构代码（保持测试通过）
```

## 🔧 详细步骤

### 🔴 RED
- 写一个失败的测试
- 测试要**明确表达意图**（不是测实现）
- 跑测试，确认它真的失败
- ❌ 不要"假装失败"（写完就 PASS 等于没测）

### 🟢 GREEN
- 写**最简**的实现让测试通过
- 不追求优雅，只求通过
- 复制粘贴也行（IMPROVE 阶段再优化）

### 🔵 IMPROVE
- 重构代码
- 消除重复
- 改善命名
- 跑全部测试，确保还通过

## 💬 提示词模板

```
请用 TDD 方式实现以下功能：

[功能需求]

要求：
1. 先写测试（RED）
2. 再写最简实现（GREEN）
3. 最后重构（IMPROVE）
4. 每步说明为什么
```

## 📌 黄金法则

- **测试要测意图，不测实现**
- **每个测试只测一件事**
- **失败信息要清晰**（"期望 X，实际 Y"）
- **覆盖率 ≥ 80%** 是目标，不是上限

## 📊 实际收益

- 开发时间：+20%（前期）
- Bug 数量：-60%（后期）
- 重构信心：高
